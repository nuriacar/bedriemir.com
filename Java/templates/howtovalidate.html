HTML5 and XHTML5

HTML5

HTML5 specification is too much forgiving about the codes of the web pages too much is allowed and well formedness is not required. This wiill lead to the sloppy codes and tendendency to more inconsistency and confusion in the page codes.

XHTML in contrary, force to well formedness, consistent codes, lowercase attributes, closing of all the tags, thus consistent and clear web pages. With XHTML when encouncountering errors one is sure they are originating from the programmer's wrong markup and not from browser unforgivness about insconsistent codes even validated as valid HTML5.

The solution is stick with the XHTML system and HTML5 specification aloows XMTHL coding style as a subset of HTML5. This is called XHTML5.This is not a new spec. because XHTML5 need not to have a new spec.everything is included in HTML. The XHTML5 is just a way coding web pages but it serving XHTML pages is different from serving HTML.

Web servers serve by default all web pages as text/html. We do not need to include, 

     <meta http-equiv ="content-type" content= "text/html";charset="utf-8" />

instead , 

     <!DOCTYPE html>
     <meta charset="utf-8"/>

will suffice for HTML for the web server defaults to "text/html" for serving the web page.

For the XHTML the situation is much more restrictive. Web server does not parse XHTML codes as HTML and serve as "text/html", but they parse as XML and serve as "application/xhtml+xml". This impose additional complexity because XML parsing is completly different in a sense that all XML files are distict and require defined parsing rules for the particular XML files. We do not wnt this. We do not want our files becomes XML. We want our pages stay as NTHL but have solid codes as XML files, the solution is Polyglott XHTML tthat is a we page which is valid HTML5 but in the same time also XHTML compliant. This is defined in ?W3C in http://dev.w3.org/html5/html-xhtml-author-guide/. Read this document and apply it carefully. The things to do are defined shortly in http://www.xmlplease.com/xhtml/xhtml5polyglot/ as:

  1. Declare namespaces explicitly.
  2. Use both the lang and the xml:lang attribute.
  3. Use <meta charset="UTF-8"/> .
  4. Use tbody or thead or tfoot in tables.
  5. When col element is used in tables, also use colgroup .
  6. Don't use noscript element.
  7. Don't start pre and textarea elements with newline.
  8. Use innerHTML property instead of document.write() .
  9. In script element, wrap JavaScript in out-commented CDATA section.
  10. Many names in SVG and one in MathML use lowerCamelCase.

Polyglot markup uses only the following named entity references:

  • amp
  • lt
  • gt
  • apos
  • quot

The entity &nbsp; can not be used in XHTML but istead we may use &#xA0; securely.

We can use http://validator.nu for validating XHTML5 pages. For validation use Text Field option and do not upload your file to to validator.nu, because MIME type declaration is still HTML5 and not XHTML but with Text Field option we can copy and paste our source codes as text field and validate against XHTML5 +SVG1.1+MathML 2.0+RDF a Lite 1.1 schema. Remember again that we do not intetend to produce an XML page and make it parse as an XML file by the browsers. We intend to produce a web page in HTML5 and we will use all the new elements and attributes of the HTML5. But, we also wish that our page codes will be as robust, well-formed, consistent and tidy as XML codes. This page is valid HTML5 but,HTML5 specification is very loose and validates the codes inacceptable by XHTML5 as valid HTML5. But, validating as valid XHTML5 brings extra strict coding style to the valid HTML5 codes. Further validation of valid HTML5 codes as XHTML5 has a value appreciated by good code crawlers. This means that you are producing tidy, acceptable and dependable codes. Also, any page validating as a valid XHTML can be converted easily to an XML page for the application of new techniques developed for XML files.
