2.6.1.2 - Cast Operator (continuing)

Widening conversion from int or long values to the data type float, may result trimming some of the least significant bits of the data , this called, the data has lost precision. One example is given below: 

package preliminary;

public class IntToFloat {

public static void main(String[] args){

 float floatValue = 0.0;
 int intValue = 13_583;

floatValue = intValue; // Widening Conversion from int to float !

System.out.println("floatValue = " + floatValue);
 } // end main

} //end class intToFloat

      The result of this program :
      floatValue = 13583.0

The result of widening conversion of int or long type data to float data type is correctly rounded value according to the IEEE 754 standard. This may be clearly seen from the result of the last program. As we can see there is no inconvenient atthe first sight in this widening conversion.

The widening conversion from int or long type data to double occurs similarly as the last conversion. The program below, shows the process: 

package preliminary;

public class IntToDouble {

public static void main(String[] args){

 double doubleValue = 0.0;
 int intValue = 26_567_444;

floatValue = intValue; // Widening Conversion from int to double !

System.out.println("doubleValue = " + doubleValue);
 } // end main

} //end class intToDouble

      The result of this program :
      doubleValue = 2.6567444E7

This widening conversion has produced the expected result for this application but, warning for loss of precision should always be kept in mind.

A value of char data type may be converted to any integral type without a problem. An example is given below: 

package preliminary;

public class CharToInt {

public static void main(String[] args){

 char characterData = 'A';
 int decimalEquivalent = characterData;

String hexVal = Integer.toHexString(decimalEquivalent);
 // String representation of a hex (base 16) equivalent of an integer of base 10 !

String hexVal = "Unicode Entry Point of " +characterData +
 " is \\u00" + hexVal;

System.out.println(hexVal);

System.out.println("Decimal Equivalent Of \\u00""  +
Integer.toHexString(decimalEquivalent)  +  "(" + characterData  + ")" +
 "="  +  decimalEquivalent);

 } // end main

} //end class CharToInt

      The result of this program :

      Unicode Entry Point of A is \u0041
      Decimal Equivalent Of \u0041(A) = 65

That is many points to clarify in the program above and the best is to go line by line. In the line, 

 char characterData = "A";

A literal from primitive char type whose value is 'A' is assigned to a variable whose data type is the primitive type char and whose name is characterData.

After the assignment is accomplished, the variable characterData holds a duality of value, either the value of the primitive char type value, either the the coresponding decimal (base 10) integer sequence value of the assigned character data.

The assignment, 

 int decimalEquivalent = characterData;

is a widening primitive type conversion from primitive data char to the primitive data type int. Actually it is more a mapping of the existing decimal integer correspondent to the integer type variable deciamalEquivalent. Losing data in that is out of question.

Now we have base 10 integer equivalent of the character data stored in a variable, but this is of little use. We use the Unicode entry points which are hex (Base 16) integers. We should find a way to realize a conversion from base 10 integers to base 16 integers. In Java integer data is manipulated by only decimal (base 10) values, and only string representations of the integers of different number bases can be obtained.

For obtaining the string representation of the base 16 equivalent of a base 10 integer we can resort to a predefiened method from the build-in Integer wrapper class who wraps the primitive data type int.

For every primitive data type, there exists a corresponding wrapper class defined in the automatically imported java.lang package. Therefore all the programs have access to the encapsulated methods in the wrapper classes. We will introdue wrapper classes after presented OOP principles. But here we are using one these methods with a pression of an immediate need. To find out about the application of this metod, we first open the online documentation from http://docs.oracle.com/javase/7/docs/ (We should memorize this!) and open the link Java SE API at the righ side of the table. In this page, we should find Integer class from all Classes column. We will tick the Integer link and in right side of the page, the methods and properties of the Integer wrapper class will be displayed. From here we will find to toHexString tab and there is the place that contains all the documentation of this method.

Access to the toHexString() method is as the following :

java.lang.Integer.toHexString(int i)

This calling system includes the package name that contains the class that in turn contains the method. This called the qualified name. There might be escape from full qualified names when the package is imported to the program. Here java.lang package is automatically imported therefore sll his public methods are accesible (visible) to the program then, we do not need to state java.lang each time we call toHexString() method.

In the documentation page, this method is introduced as:

public static String toHexString(int i)

The meaning of public is that this class is accessible from all classes that can access Integer() wrapper class.

The meaning of static is that this method belongs to the class definition itself ans can only be called as Integer.toHexString();

String is the data type of the value that this method returns.

Finally, the method is called with a single argument with type int. The documentation indicates that the method returns a string representation of the integer argument as an unsigned integer in base 16. The unsigned integer value is the argument plus 232 if the argument is negative; otherwise, it is equal to the argument. This value is converted to a string of ASCII digits in hexadecimal (base 16) with no extra leading 0s. If the unsigned magnitude is zero, it is represented by a single zero character '0' ('\u0030'); otherwise, the first character of the representation of the unsigned magnitude will not be the zero character.

In simple words, the argument should be an unsigned integer literal, the returning value is the string representation of the equivalent of the argument in unsigned integers in base 16.

From the information above, the program step, 

String hexVal = Integer.toHexString(decimalEquivalent);

will create a variable named hexVal in type String, whose initial value is the equivalent of the value of the variable decimalEquivalent in hex.

The remaining of the program is nothing than the arrangements to get program results with explicative messages. Note the escape in slash character as "\\u00".

The result of the exectuion of the CharToInt() class, show that the Unicode entry points may be calculated from this program but only the values whose hex equivalents can be expressed with two digits.

I owe you an apology for the application of a method that will be introduced in forthcoming chapters. But the usage of the build-in classes is well documented and we can use some of them when the need arise, before their introduction in this textbook.
