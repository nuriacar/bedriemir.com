2.6.1.1 - Assignment Operator (continuing)

Let us experience about what will happen if we try to assign overfowing and underflowing data to a variable. We will examine the behavior of int type data in the example but the result is similar for other primitive types.

The program below will give us a clearer understanding about the value assigned to a int type variables when the value to be assigned is graeter than the defined maximum value for the primitive int data type. 

package preliminary;

public class Overflow {

public static void main(String[] args){

 int upperLimit = 2_147_483_647;
 int oneMore = upperLimit + 1;
System.out.println(" 2_147_483_647 + 1 = " + oneMore);
 } // end main

} //end class Overflow

The result of this program : 

2_147_483_647 + 1 = -2147483648

The result clearly indicates that a wrong value is assigned to the variable, without a single warning that an erroneous value is assigned to the variable. Neither a rejection to the compilation, nor even a runtime error. Just silently (I could say sneaky) assignment of the wrong value.

The program below, covers the same phenomenon in case of underflow, again with int type of variable assignment. 

package preliminary;

public class Underflow {

public static void main(String[] args){

 int lowerLimit = -2_147_483_648;
 int oneLess = lowerLimit - 1;
System.out.println(" -2_147_483_648 - 1 = " , oneLess);
 } // end main

} //end class Underflow

The result of this program : 

-2_147_483_648 - 1 = 2147483647

Again an assignment of a wrong value, without a single warning.

The program below, show the same situation in case of double data type: 

package preliminary;

public class OverflowInDouble {

public static void main(String[] args){

 double over = 1.7E308;
 over = over * 1.1;
System.out.println(over);

 double under = 1.0E-323;
 under = under / 10.0;
System.out.println(under);
 } // end main

} //end class Underflow

The result of this program : 

Infinity
0.0

This time at least, some nonnumeric values whose values represent the situation. Again no warning. Neither a compile time, nor a runtime exception is raised in case of assignment of underflowing or overflowing values to the variables.

Since in case of assigning overflowed or underflowed values to the variables, wrong values are assigned and an exception is not raised, the programmer may even not be aware that an unexpected an erroneous action has incidently happened in the execution of the program.

As we can see from these examples, we should not let our assignment operation realize with an erroneous result. Over or underflowing values should not be allowed to be assigned to the variables. Preventing this, may be difficult or easy depending on the situations. If the program is not receveing data from the outside, it may be easier to control the results by comparing with the known values and inspection of program steps if some kind of anomaly is existing. If the cause is overflowing values, we can promote the data type, for example from int to long, even to non overflowing java.math.BigInteger. That will remedy the issue and the calculations may reliably continue to deliver the results.

The situation worsens with the interactively driven programs, where sometimes huge amount of external data is transfered and processed. These should reliably deliver results and for reaching this, all the data must be validated before further processing. Incoming data must be validated first with data validation routines and any unreliable data must be discarded tracked to source which delivers it and the causes of the bad data revealed. This is not an easily implementable methodology but can be mastered with time and effort. One more reason for being proficient in the subject.

Especially for programs open to the outside world, receiving huge amount of external data, processing and retransmitting them, the results of delivering wrong values is detrimental. It should be avoided for any cost. I can not imagine the results of supplying unreliable data to any innocent outside receiver. This is an overly serious situation.

When we take things seriously, we can always find a solution. To my opinion, especially for bigger and interactive programs, some automated data validation routines are necessary. Ours is ready, but we can not present and explain here. We should wait for having more understanding about the Java technology.
