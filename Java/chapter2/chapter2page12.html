2.6.1.1 - Assignment Operator (continuing)

In Java, every primitive data type has a validity range. If a value of a primitive type variable is lower then the minimum value defined for this type, it is said there is an underflow. Respectively, if the value of a variable exceeds the maximum value defined for this type, it is said that there is an overflow. The sketch below, may be useful to clarify the situation.

underflow<valid values<overflow

For example, the validity range for int type variables is: 

   underflow<-2_147_483_648<int type values<2_147_483_647<overflow

The maximum and minimum values of a primitive data type is defined according to the width of the memory place defined for this data type. If the value to be stored in that memory portion exceeds the housing capacity of this predefined and unchangeable memory width, an overflow or underflow occurs.

When in an assignment statement, the programmer forces a variable with a determined data type, to a value that the memory width defined for this data type can not hold, there might be many alternatives to choose for a compiler. But to my point of view, Java compiler takes the worst of actions it produces a synthetic wrong value that can fit to the reserved memory location, store this wrong value to the memory place and finish the assignment with wrong value without giving a single clue to the user. In case of overflow/underflow, Java compiler discards the higher order bits, preserve lower order bits, generates an erroneous shrunk value and allocates it to the memory place that can be accessed by the variable. During this operation, the most significant bit (MSD) also is changed and the generated wrong value is of the opposite sign of the true value. The assignment statement terminates successifully (!) , by assigning a wrong value to the variable, but without any indication of an exception. If you are not aware (and normally there no way of being aware!), every time you call this variable a wrong value will be returned. This more than a scandal, but this is the way things are going on.

Theoretically, "An exception occurence synonymous of delivering the standard service specified for a program." as described by Flaviu Christian. Practically, it is synonymous of a program error. In computing words, when an abnormal condition is encountered in a program, it is said that an exception is raised, or an exception is thrown. If an exception is thrown while in compiling, it is called a compile time exception. If it occurs while running the program, it is called a runtime exception. In Java, exceptions may be handled, thus preventing the abrupt ending of the program when an exception is thrown. If an exception is thrown and it is not properly handled, the error will propagate to the higher areas of the program and most of the time the program will end abrubtly. When a program ends abruply and inadvertently, we say that the program is crashed. When a program crashes, it may also trigger the events that may also crash the whole operating system and this is some sort of disaster. In Java, when there is a detectable error in the program, Java compiler stops compiling and issues a message indicating why the compilation was not succesiful. If an exception is thrown in the runtime, program execution stops and a runtime exception message is generated.
