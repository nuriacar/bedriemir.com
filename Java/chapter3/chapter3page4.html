3.6.2 - Access Modifiers.

Access modifiers (or access specifiers) act upon class definitions to organize their accessibility and the nature of the class being defined. In the meantime, accessibiliy of internal members of a class can be regulated via access modifiers.These two application areas are different from each other and we will cover the impact of access modifiers to the classes first.

Access modifiers act upon a class definition by pre or post specifiying the nature of the class being defined. They are all optative and nothing may be specified at all. 

[]pre specifiers] ClassName [post specifiers]

Access modifiers acting as pre specifiers are :

  • public
  • Nothing is specified
  • abstract
  • final

Public classes may be accessed from everyone and from everywhere. To access this class from other packages, first the package in which the class is defined, must be imported to class claiming the access.

Classes are best defined as public classes. This grants their universal access. Classes are made for being useful. There in no reason to restrict their access only from their originating package, except in rare cases where limiting reasons are obvious only to the designer of this class.

When not any access specifier is pronounced, the compiler defaults this class as only be accessible from within the package it is defined. This extra restriction may not be necessary. Better to define as public, when no absolute necessities exist for its restriction.

The abstract acess modifier may be seen weird in the first sight.The class definition is already an abstracted data type (ADT) and the reasons of application a new abstraction may not be well understood. In fact the abstract access modifier has no relation with the philosophical abstraction elaborated when defining a class structure. The abstract access modifier only indicates that the class being defined will only include general definition about what the actual class structure will be and no actual program steps will be declared for the moment being.

We should differentiate between,

  • Concrete Classes
  • Abstract Classes

Concrete classes are declared with all the components defined. Nothing left to the future. Every declared metods have their bodies concretly presented. This the way we took with a definitions of all the smple classesso far. Including the last one namely Arithmetical class. A concrete class may be instantanised (producing new instances of this class) and extended (producing subclasses of this class)

An abstract class in contrast has only the definitions and not how they will be achieved. They are incomplete classes, they have definitions but not implementations. Abstract classes may not be instantanied, because nothing is clear by now. The definition lacks implementation and new examplaires may produced from only well defined unambigous models. An abstract class is only an idea. Everybody can realize a concrete model with his own understanding from this idea. Once a concrete definition is realised, millions of objects may be produced from this concrete model. not before. We will see an example of an abstract class when we will study the access modifiers of the class members.

One way of concretisation of an abstract method, is to provide implementation trough extension. That is an abstract method may be subclassed with implementation of concrete program lines to his abstract methods. This can be achieved in one class, but normally one class defines the abstract methods and many other subclasses implements this ancestor abstract class. An example is given below: 

package objects;

class AbstractArithmetical {
static double multiply ( firstElement , secondElement) ;
}

A final class is a class that may not extended. That is, no subclasses may be derived from this class.

A post-specifier extends ParentClass reveals that the classs being defined is extending the ParentClass. This means, we are defining a subclass of the ParentClass. The ParentClass must be extensible of course. We will see examples of hierarchical classses when studying hierarchical systems.

A post-specifier implements SomeAbstractClass[,SomeOtherAbstractClass, ...], is signalling that the class being defined is implementing abstract methods defined in a given Abstract Classes List. Examples will be presented in the explanation of the abstract class members.
